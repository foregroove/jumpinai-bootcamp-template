## Day 2 §4 Phase 3 — アプリ実装

### Step 1: 雛形作成（15分）

```
このプロジェクトの src/script.py に、以下を満たす雛形を作って。

要件:
- コマンドライン引数で1つの画像ファイルパスを受け取る
- 画像が存在しない場合はエラーログを出して終了
- main() 関数で全体処理をまとめる
- if __name__ == "__main__" パターンで起動
- logging を使ったログ出力（print禁止）
- 型ヒントを付ける

CLAUDE.md の技術スタックと命名規則に従って。
requirements.txt も同時に作成して。
```

### Step 2 直前: APIキー設定（3分）

これはプロンプトではなく、手動操作です：

> **APIキーの入手方法：**
> - **🧑‍💻 セルフラーニング・1on1**: `docs/pre-bootcamp-kit.md` §0-6「Anthropic APIキー取得」を完了させ、取得した `sk-ant-...` を使用
> - **🏫 Workshop参加**: 講座開始3日前に個別配布されたキーを使用
>
> どちらのモードでも、以下の手動操作は同じです。

```bash
# ターミナルで実行
cp .env.example .env
# エディタで .env を開いて、準備した sk-ant-... を貼り付け

# 確認
cat .env | grep ANTHROPIC_API_KEY
```

### Step 2: Vision API で画像読み込み（25分）

```
src/script.py を拡張して、Claude Vision API でレシート画像を読み込む処理を実装して。

要件:
- python-dotenv で .env から ANTHROPIC_API_KEY を読み込む
- ANTHROPIC_API_KEY が "your_key_here" のままならエラーで終了
- anthropic SDK を使用
- claude-sonnet-4-6 モデル
- 画像を base64 エンコードして送信
- 画像形式は jpg/jpeg/png に対応すること（拡張子から media_type を判定）
- AIへのプロンプトは「このレシートから日付・店名・金額を読み取ってください。
  日付はYYYY-MM-DD形式で。」
- レスポンスを logging.info で出力
- API エラー時は logging.error で記録して終了

CLAUDE.md のエラーハンドリング規約に従って。
```

**※ v0.2 改訂点（#2, #3）：媒体タイプ対応とAPIキー未設定検知を要件に追加**

### Step 3: JSON構造化（20分）

```
src/script.py を拡張して、Vision API のレスポンスを JSON 構造化する処理を追加して。

要件:
- AIへのプロンプトを変更：「以下のJSONフォーマットで返してください」を追加
  {"date": "YYYY-MM-DD", "store": "店名", "amount": 金額（整数）, "category": "経費科目"}
- 経費科目は CLAUDE.md の5分類のいずれか（会議費/消耗品費/旅費交通費/新聞図書費/雑費）
- AIのレスポンスから JSON 部分のみを抽出してパース
- パース後のバリデーション：
  - 必須キー（date, store, amount, category）の存在確認
  - amount は整数化（文字列で来た場合も対応）
  - category が5分類のいずれかであることを検証（範囲外なら "雑費" に丸める）
- パース失敗時は logging.error で生データを記録
- 成功時は dict として main() に返す
```

**※ v0.2 改訂点（#4）：バリデーション要件を明示**

> ⚠ ここでカテゴリ判定がブレる例：スーパー（マルエツ等）や 100 円ショップは
> シンプルなプロンプトだと「消耗品費」と判定されがちです。
> このブレ自体が **§5-1 で CLAUDE.md にルール詳細を書く動機** になります。

### Step 4: CSV出力（20分）

```
src/script.py を拡張して、構造化データを CSV ファイルに出力する処理を追加して。

要件:
- pandas を使用
- 出力先は output/expenses_YYYYMMDD_HHMMSS.csv（タイムスタンプ付き）
- 列順は CLAUDE.md の通り：日付,店名,金額,勘定科目
- UTF-8（BOMなし）
- 既存ファイルに追記せず、毎回新規作成
- output/ ディレクトリがなければ作成
- 完了時に logging.info で出力先パスを表示
```

> ※ §7「発展課題」候補：1枚 = 1CSV ではなく `python src/script.py samples/`
> でディレクトリ一括処理 → 1ファイル化（#7）

## Day 2 §5 — 三種の神器

### 章の前に：受講者に伝えるべきメッセージ（重要）

§5-1 / §5-2 のハンズオンで `CLAUDE.md` や `SKILL.md` を拡張しても、
**現状の `src/script.py` の判定精度はその場では変わりません**。

理由：
- `CLAUDE.md` は **Claude Code（開発支援AI）が読む設計ドキュメント**
- `script.py` 内の `VISION_PROMPT` には自動で伝播しない
- 分類を行う runtime AI は `CLAUDE.md` の存在を知らない

**§5-1 / §5-2 で身につくのは「設計ドキュメントを構造化する習慣」と「再利用可能な判断基準を切り出す力」。**
実精度を変える体験は **§5-3 で `SKILL.md` を runtime に注入する改修** で得ます。

この前提を講座冒頭で明示することで、受講者の「やったのに何も変わらない」混乱を防ぎます。

---

### §5-1 ハンズオン: CLAUDE.md に経費分類ルール詳細を追記（10分）

**※ 重要：このプロジェクトは事前配布時から `CLAUDE.md` が存在します。"作る" ではなく "拡張する" です。**

```
このプロジェクトの CLAUDE.md を拡張して、経費科目の判定ルールをより詳細に書いてください。

現状の5分類（会議費／消耗品費／旅費交通費／新聞図書費／雑費）はそのまま、
各分類の判定基準を3〜5項目ずつ追加して。

例えば「会議費」なら：
- カフェ・喫茶店・ファストフード（自分の食事は除く）
- 取引先との飲食代
- 打合せ用の手土産・茶菓子
- ...

実例ベースで、AIが迷いそうなグレーゾーンを潰すような書き方で。
特に「スーパー」「100円ショップ」「コンビニ」などの判定指針を明示してください。
（§4 Step 3 でこれらが "消耗品費" 一律判定だった問題を、ルール記述で解消します）
```

**※ v0.2 改訂点（#8）：「拡張する」を明示、Step 3 のブレ事例との接続を強化**

### §5-2 ハンズオン: 経費分類ルールをSKILL.md化（12〜15分）

```
.claude/skills/expense-categorize/SKILL.md を作成して、CLAUDE.md の経費科目判定ルールを
独立したスキルとして切り出して。

要件:
- YAML frontmatter（name, description）必須
- description には「いつこのスキルが呼び出されるべきか」を具体的に書く
  例：「レシート情報から経費科目（会議費／消耗品費／旅費交通費／新聞図書費／雑費）を
       判定する必要がある時に使う」
- 本文には判定ルールと、判定の難しい3例の解説を含める
- 2500語以下に収める

完成したら、CLAUDE.md からは詳細ルール部分を削除し、「経費科目分類は expense-categorize 
スキルを参照」と1行記述して整理して。
```

**※ v0.2 改訂点（#10）：時間枠 12-15分に拡張、文字数制約を 1500-2000 → 2500語に緩和**

### §5-3 ハンズオン: SKILL.md を runtime prompt に注入する（15分）★ NEW in v0.2

```
src/script.py を拡張して、起動時に .claude/skills/expense-categorize/SKILL.md の本文を
読み込み、Vision API への分類プロンプトに自動的に組み込む実装を追加して。

要件:
- スクリプト起動時に SKILL.md ファイルが存在すれば本文（YAML frontmatter は除外）を読み込む
- 読み込んだ内容を VISION_PROMPT のサフィックスとして連結する：
  例: VISION_PROMPT + "\n\n## 分類ルール詳細\n" + SKILL.md本文
- SKILL.md が存在しない or 読み込み失敗時は警告ログを出して、従来通り素のプロンプトで動く
- ロード結果は logging.info で「skill loaded: ... bytes」と記録

実装後、§4 Step 3 で "消耗品費" と誤判定されたサンプル（スーパー／100円ショップ）を
再実行して、判定が正しくなるか確認してください。
```

**※ v0.2 新規追加（#9 b案）：これが §5 における「実精度向上の唯一のハンズオン」**

> 講師ノート：このハンズオンが §5 のクライマックス。「設計ドキュメント」を「動く仕組み」に
> 接続する瞬間で、受講者の理解が一気に深まる。

### §5-4: MCP の紹介（旧 §5-3）

実装ハンズオンは行わず、概念紹介のみ。
**注：v0.6 までは §5-3 だったが、v0.7 で runtime injection が §5-3 になり MCP が §5-4 に押し下げ。**

## Day 2 §6 — 動いた、で終わらせない

> **重要：このプロジェクトは事前整備済みのため、§6 全体が「確認パート」になります。**
>
> 受講者がゼロから git init / README作成 / .env作成 をするのではなく、
> 「すでに整っている状態を**自分で確認**して、なぜそうなっているかを**説明できる**」
> ことを目指します。これが本講座の §6 の学習目標です。
>
> **「動いた、で終わらせない」習慣 = 整っている状態を読み解ける力**、と再定義しています。

### §6-1 ハンズオン: Git 管理状況の確認（5分）

**※ v0.2 改訂点（#13）：事前 git init 済みのため「確認モード」に変更**

```
このプロジェクトの Git 管理状況を確認して、以下を報告して：

1. git init されているか（.git ディレクトリの存在）
2. これまでの commit 履歴（git log --oneline -10）
3. 現在のブランチと、リモート追跡状況
4. .gitignore で何を除外しているか（特に .env / output / samples の扱い）
5. 追跡対象ファイルの一覧（git ls-files）

それぞれが「動いた、で終わらせない」習慣のどの要素に対応するかも併せて説明して。
```

### §6-2 ハンズオン: README の点検（5分）

```
このプロジェクトの README.md を読んで、以下を判定して：

1. 5セクション構成（概要／セットアップ／使い方／設計判断／既知の制限）が揃っているか
2. 別 PC を渡された他人が、5分で動かせる粒度で書かれているか
3. 不足があれば、追加すべき内容を提案して
4. このプロジェクトの README は CLAUDE.md とどう違うか／なぜ両方あるのか説明して
```

### §6-3 ハンズオン: シークレット三層防御の確認（5分）

```
このプロジェクトのシークレット管理が三層防御になっているか確認して、結果を報告して。

確認項目:
1. コードに APIキーが直書きされていないか（grep で確認）
2. .env が .gitignore に入っているか
3. .env.example が commit されているか
4. dotenv で .env を読み込んでいるか

問題があれば修正して、commit までやって。問題なければ「三層防御OK」と報告して。
```

### §6-4: デプロイ前チェック（5分・実行ハンズオン）

スライド §6-4 のチェックリスト5項目を、AI に依頼する形：

```
このプロジェクトを GitHub 公開する前のデプロイ前チェックを実施して、結果を報告して。

チェック項目:
1. .env / 個人情報がcommitされていないか（git ls-files で確認）
2. README が他人に通じる内容か
3. エラー時の挙動が試されているか（壊れた画像 / API失敗 / ネット切断）
4. 依存パッケージが requirements.txt に固定されているか
5. 動いた状態で commit 済みか（git status がclean）

各項目の結果と、修正が必要なら修正案を出して。
```

## トラブルシューティング用プロンプト

### エラーが出た時の質問テンプレ（§7-3）

```
[何をしようとしているか]
> python src/script.py samples/receipt_001.jpg を実行している

[何が起きたか]
> 以下のエラーが出る:
[ここにエラー全文を貼る]

[どこまで試したか]
> 1. cat .env | grep ANTHROPIC_API_KEY で値が見えることは確認済み
> 2. anthropic ライブラリは pip list で確認済み
> 3. ...

[聞きたいこと]
原因と次の確認手順を教えて。
```

### git で詰まった時

```
git で困っている。今の状態を git status と git log --oneline -5 で確認して、
何が起きているか説明して。次にやるべきことも提案して。
```

### step-N-done に逃げる時

```bash
# プロンプトではなくコマンド
git stash                    # 自分の変更を退避
git fetch                    # リモート最新化
git checkout step-3-done    # 該当ステップの完了状態にジャンプ
# Claude Code に状態を確認してもらう
```
