## Day 2 §4 Phase 3 — アプリ実装

### Step 1: 雛形作成（15分）

```
このプロジェクトの src/script.py に、以下を満たす雛形を作って。

要件:
- コマンドライン引数で1つの画像ファイルパスを受け取る
- 画像が存在しない場合はエラーログを出して終了
- main() 関数で全体処理をまとめる
- if __name__ == "__main__" パターンで起動
- logging を使ったログ出力（print禁止）
- 型ヒントを付ける

CLAUDE.md の技術スタックと命名規則に従って。
requirements.txt も同時に作成して。
```

### Step 2 直前: APIキー設定（3分）

これはプロンプトではなく、手動操作です：

```bash
# ターミナルで実行
cp .env.example .env
# エディタで .env を開いて、配布された sk-ant-... を貼り付け

# 確認
cat .env | grep ANTHROPIC_API_KEY
```

### Step 2: Vision API で画像読み込み（25分）

```
src/script.py を拡張して、Claude Vision API でレシート画像を読み込む処理を実装して。

要件:
- python-dotenv で .env から ANTHROPIC_API_KEY を読み込む
- anthropic SDK を使用
- claude-sonnet-4-6 モデル
- 画像を base64 エンコードして送信
- AIへのプロンプトは「このレシートから日付・店名・金額を読み取ってください。日付はYYYY-MM-DD形式で。」
- レスポンスを logging.info で出力
- API エラー時は logging.error で記録して終了

CLAUDE.md のエラーハンドリング規約に従って。
```

### Step 3: JSON構造化（20分）

```
src/script.py を拡張して、Vision API のレスポンスを JSON 構造化する処理を追加して。

要件:
- AIへのプロンプトを変更：「以下のJSONフォーマットで返してください」を追加
  {"date": "YYYY-MM-DD", "store": "店名", "amount": 金額（整数）, "category": "経費科目"}
- 経費科目は CLAUDE.md の5分類のいずれか（会議費/消耗品費/旅費交通費/新聞図書費/雑費）
- AIのレスポンスから JSON 部分のみを抽出してパース
- パース失敗時は logging.error で生データを記録
- 成功時は dict として main() に返す
```

### Step 4: CSV出力（20分）

```
src/script.py を拡張して、構造化データを CSV ファイルに出力する処理を追加して。

要件:
- pandas を使用
- 出力先は output/expenses_YYYYMMDD_HHMMSS.csv（タイムスタンプ付き）
- 列順は CLAUDE.md の通り：日付,店名,金額,勘定科目
- UTF-8（BOMなし）
- 既存ファイルに追記せず、毎回新規作成
- output/ ディレクトリがなければ作成
- 完了時に logging.info で出力先パスを表示
```

## Day 2 §5 — 三種の神器

### §5-1 ハンズオン: CLAUDE.md に経費分類ルール詳細を追記（10分）

```
CLAUDE.md を拡張して、経費科目の判定ルールをより詳細に書いてください。

現状の5分類はそのまま、各分類の判定基準を3〜5項目ずつ追加して。
例えば「会議費」なら：
- カフェ・喫茶店・ファストフード（自分の食事は除く）
- 取引先との飲食代
- 打合せ用の手土産・茶菓子
- ...

実例ベースで、AIが迷いそうなグレーゾーンを潰すような書き方で。
```

### §5-2 ハンズオン: 経費分類ルールをSKILL.md化（10分）

```
.claude/skills/expense-categorize/SKILL.md を作成して、CLAUDE.md の経費科目判定ルールを
独立したスキルとして切り出して。

要件:
- YAML frontmatter（name, description）必須
- description には「いつこのスキルが呼び出されるべきか」を具体的に書く
  例：「レシート情報から経費科目（会議費／消耗品費／旅費交通費／新聞図書費／雑費）を
       判定する必要がある時に使う」
- 本文には判定ルールと、判定の難しい3例の解説を含める
- 1500〜2000語以下に収める

完成したら、CLAUDE.md からは詳細ルール部分を削除し、「経費科目分類は expense-categorize 
スキルを参照」と1行記述して整理して。
```

## Day 2 §6 — 動いた、で終わらせない

### §6-1 ハンズオン: Gitリポジトリ化（5分）

```
このプロジェクトをGit管理にしてください。

要件:
- .gitignore は既存のものをそのまま使う
- 最初のコミットメッセージは "first working version: receipt to CSV"
- git log で確認できる状態にする
```

### §6-2 ハンズオン: README追加（5分）

```
CLAUDE.md の内容を踏まえて、README.md を書いてください。

5セクション構成で:
1. 概要（1〜2行）
2. セットアップ（コマンド付き）
3. 使い方（実行例付き）
4. 設計判断（CSVから始めた理由、Vision API選択理由など）
5. 既知の制限（できないこと）

完成したら git add README.md && git commit -m "add README" まで実行して。
```

### §6-3 ハンズオン: .env + .gitignore セットアップ（5分）

このプロジェクトは事前配布時から .env / .gitignore / .env.example が整っているので、
このハンズオンは「現状を確認する」流れになります：

```
このプロジェクトのシークレット管理が三層防御になっているか確認して、結果を報告して。

確認項目:
1. コードに APIキーが直書きされていないか（grep で確認）
2. .env が .gitignore に入っているか
3. .env.example が commit されているか
4. dotenv で .env を読み込んでいるか

問題があれば修正して、commit までやって。
```

### §6-4: デプロイ前チェック

スライド §6-4 のチェックリスト5項目を、AI に依頼する形：

```
このプロジェクトをGitHub公開する前のデプロイ前チェックを実施して、結果を報告して。

チェック項目:
1. .env / 個人情報がcommitされていないか（git ls-files で確認）
2. README が他人に通じる内容か
3. エラー時の挙動が試されているか
4. 依存パッケージが requirements.txt に固定されているか
5. 動いた状態で commit 済みか（git status がclean）

各項目の結果と、修正が必要なら修正案を出して。
```

## トラブルシューティング用プロンプト

### エラーが出た時の質問テンプレ（§7-3）

```
[何をしようとしているか]
> python src/script.py samples/receipt_001.jpg を実行している

[何が起きたか]
> 以下のエラーが出る:
[ここにエラー全文を貼る]

[どこまで試したか]
> 1. cat .env | grep ANTHROPIC_API_KEY で値が見えることは確認済み
> 2. anthropic ライブラリは pip list で確認済み
> 3. ...

[聞きたいこと]
原因と次の確認手順を教えて。
```

### git で詰まった時

```
git で困っている。今の状態を git status と git log --oneline -5 で確認して、
何が起きているか説明して。次にやるべきことも提案して。
```

### step-N-done に逃げる時

```bash
# プロンプトではなくコマンド
git stash                    # 自分の変更を退避
git fetch                    # リモート最新化
git checkout step-3-done    # 該当ステップの完了状態にジャンプ
# Claude Code に状態を確認してもらう
```
